# 煤矿安全综合监控平台原型产品需求文档（PRD）

> 项目：煤矿安全综合监控平台

> 文档落点：`src/resources/`

## 一、文档目的

本文档用于沉淀当前“煤矿安全综合监控平台”原型项目的产品定位、范围边界、模块结构、原型策略与阶段验收口径，作为后续页面复刻、模块优化、需求补充和评审沟通的统一依据。

当前项目以“先复刻底稿、再逐页优化”为主线，不追求一次性铺满全系统，而是围绕已提供截图、页面链接和明确需求的模块逐步扩展。

## 二、项目背景

现阶段目标不是从零定义一个全新平台，而是在现有业务系统基础上，构建一个可演示、可讨论、可持续迭代的前端原型底座，用于承接以下工作：

- 复刻既有煤矿监管系统页面，形成高保真原型底稿。
- 在用户确认“像不像”之后，再继续做信息层级、交互路径和业务表达优化。
- 将零散页面需求、页面决策和原型调整结果沉淀为长期可复用的项目资料。

该项目当前默认显示名为：**煤矿安全综合监控平台**。

## 三、产品目标

### 3.1 总体目标

- 建立统一的系统原型底座，承接后续模块级页面复刻和迭代。
- 通过原型快速展示监管平台典型页面结构、操作路径和信息组织方式。
- 为客户沟通、内部评审和后续需求说明提供可视化依据。

### 3.2 本阶段目标

- 完成统一顶部导航、左侧菜单、内容容器和通用页面模板搭建。
- 完成若干代表性页面复刻，验证底座复用能力。
- 在短信管理模块内完成一轮相对完整的模块复刻与迭代，形成可持续扩展的页面组。

## 四、目标用户与核心场景

### 4.1 目标用户

本原型面向以下典型角色：

- 监管侧业务人员：查看报警、企业档案、视频信息、短信发送及接收配置等业务页面。
- 项目沟通与评审人员：用于讲解页面结构、业务路径和后续优化方向。
- 后续设计与研发协同人员：以原型作为功能边界和交互意图的参考底稿。

### 4.2 核心场景

- 基于现有系统截图或线上页面，对指定模块进行高保真复刻。
- 对已复刻页面进行结构优化、字段增减、只读化调整或交互补全。
- 在固定系统底座内新增某个模块页面，而不是每次单独新建一个完全独立的原型。

## 五、产品范围

### 5.1 当前纳入范围

当前项目范围包括三类内容：

1. 系统级复用底座  
   包括顶部导航、左侧菜单、通用列表页、详情页、查询区、工具条、表格、分页、占位页等。

2. 已复刻示例模块  
   作为系统底座验证页面，当前已覆盖：
   - 监测预警 / 报警处置
   - 基本信息 / 企业档案查询
   - 视频集中管理平台 / 视频信息详情

3. 已重点扩展模块  
   当前已明确进入持续迭代的模块为：
   - 短信管理

### 5.2 当前不纳入范围

- 不提前对所有顶部模块做全量高保真复刻。
- 不在没有截图、链接或明确需求的情况下凭空扩展完整业务流程。
- 不在本阶段实现真实后端接口、真实权限逻辑或真实数据计算。
- 不把原型项目升级为正式生产系统前端工程。

## 八、原型实现原则

### 8.1 先复刻，后优化

当用户提供页面截图或链接时，优先按现有系统做高保真复刻，先确认页面“像不像”；确认后再讨论是否需要调整信息层级、视觉密度、交互路径或业务表达。

### 8.2 优先复用底座

顶部导航、左侧菜单、内容容器、查询区、工具条、表格、分页、详情分组、弹窗/抽屉和占位页应尽量复用，不为单个页面重复造壳。

### 8.3 按模块逐页推进

当前项目采用“指定页面驱动”的建设方式：做哪个页面，就先复刻哪个页面；未明确的模块继续保持占位状态。

### 8.4 文档与原型同步沉淀

当页面范围、结构、只读/可编辑口径、字段取舍或菜单入口发生变化时，应同步把确认结果沉淀到：

- `src/prototypes/<prototype-name>/.spec/`
- `src/resources/`

避免只改原型、不留可追溯决策。

## 九、内容与数据约束

### 9.1 数据来源口径

当前原型中的表格、树节点、详情字段和统计内容，默认以演示数据和页面结构表达为主，用于还原页面语义和信息层级，不视为真实业务数据库结果。

### 9.2 内容来源优先级

当前项目的页面内容来源优先级为：

1. 用户当前消息、截图、链接和明确口述要求
2. 仓库内现有原型代码和 `.spec` 决策快照
3. `src/resources/` 下已沉淀的需求说明与资料
4. 系统底座既有样式、组件和页面结构

### 9.3 异常与边界

对于缺少截图、缺少字段定义、缺少操作页原貌或会明显影响范围判断的页面，不直接臆造完整方案，应先停留在占位页或已知结构级别，待补充资料后继续推进。

## 十、验收标准

### 10.1 项目级验收

- 原型显示名保持为“煤矿安全综合监控平台”。
- 原型主入口保持在 `src/prototypes/system-replica-base/`。
- 系统底座具备顶部导航、左侧菜单和统一内容容器。
- 已确认页面可在底座内访问，不因新增页面破坏整体壳结构。

### 10.2 页面级验收

- 已复刻页面在视觉结构上与既有系统保持一致风格。
- 已迭代页面能准确体现本轮确认过的字段、分组、入口和只读/可编辑边界。
- 未纳入范围页面明确保持占位，不出现与真实需求冲突的臆造内容。

### 10.3 文档级验收

- 每轮关键决策有对应 `.spec` 快照或需求说明沉淀。
- 项目资料统一落在 `src/resources/`，原型决策统一落在对应 `.spec/`。
- 交付内容能支撑后续继续复刻、继续优化或继续补写某个模块 PRD。

## 十一、后续建议

后续可按以下顺序继续沉淀：

1. 继续围绕 `system-replica-base` 补齐明确指定模块页面。
2. 针对每个已进入迭代的模块，在 `src/resources/` 下补齐模块级 PRD。
3. 在设计基底明确后，再补写对应页面或模块的 UI Review。
4. 当某个模块需求足够稳定后，再从“复刻底稿”进入“结构优化版”阶段。

## 十二、本轮结论

当前项目已经具备“系统底座 + 示例页面 + 短信管理模块完整页面组”的原型基础，适合继续采用“提供参考页面 -> 复刻底稿 -> 确认像不像 -> 补需求说明 -> 继续优化”的方式迭代推进。

本 PRD 作为项目级底稿，后续新增模块或页面时，可在此基础上继续拆分更细的模块级需求说明。
